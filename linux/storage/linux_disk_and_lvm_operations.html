Both workflows below need filesystem-aware commands at the resize/repair stage. Pick the row matching the LV's filesystem — confirm with blkid or findmnt if unsure.

Filesystem Typical distro Grow Repair UUID regen

XFS quirk: xfs_growfs takes a mountpoint, not a device — the FS must be mounted to grow it. resize2fs takes either, and works mounted or unmounted.

Common Scan for the new disk
01 Trigger SCSI rescan and identify the new device DISCOVER ▶
Quick rescan # Tells the kernel to re-enumerate every SCSI host adapter echo "- - -" | sudo tee /sys/class/scsi_host/host*/scan lsblk Verbose rescan — list /dev/sd* in one pass for host in /sys/class/scsi_host/*; do echo "- - -" | sudo tee $host/scan ls /dev/sd* done

The new disk appears at the bottom of lsblk with no partition or mount entries. Examples below assume /dev/sdf (extend) or /dev/sdb (recovery) — substitute the actual device returned by your rescan before running anything destructive.

Workflow A Extend rootvg with a new disk
02 Add the new disk to the volume group EXTEND ▶
# Initialise the new disk as an LVM physical volume pvcreate /dev/sdf # Confirm the existing volume group name vgs # Example output: # VG #PV #LV #SN Attr VSize VFree # vg 3 1 0 wz--n- 233.51g <34.00g # Extend the VG with the new PV vgextend vg /dev/sdf

Note: the VG name in the example above is vg — your real name will differ (commonly rootvg, system, or distro-specific). Always read the actual name from vgs before running vgextend.

03 Extend the logical volume and grow the filesystem RESIZE ▶
EXT4 — Debian / Ubuntu lvextend -l +100%FREE /dev/mapper/vg-root resize2fs /dev/mapper/vg-root XFS — RHEL / Rocky / Alma / OL lvextend -l +100%FREE /dev/mapper/vg-root xfs_growfs /dev/mapper/vg-root reiserfs — SLES lvextend -l +100%FREE /dev/mapper/vg-root resize_reiserfs /dev/mapper/vg-root

+100%FREE consumes everything available in the VG. To grow by a fixed amount, use lvextend -L +20G (absolute growth) instead. Always include the + sign — bare -L 20G sets the absolute size and will shrink a larger LV.

04 Verify VERIFY ▶
# Block-level view — new size should be reflected on the LV lsblk # PV-level view — new disk visible, allocated to the VG pvs # Filesystem-level view — used / available figures should reflect the new size df -h
Workflow B Attach & recover an existing LVM disk
05 Import the foreign VG under a fresh name IMPORT ▶
# Use vgimportclone — assigns a new VG name and renames PVs accordingly. # The filter clause stops LVM picking up the host's own PVs during import. vgimportclone --basevgname newvg2 \ --config 'devices { filter = [ "a|/dev/sdb|", "r|.*|" ] }' \ /dev/sdb2 /dev/sdb3

Why vgimportclone and not vgchange -ay directly: the recovered disk almost certainly carries the same VG name and UUIDs as another active VG (often rootvg). Activating it as-is risks LVM picking the wrong PVs at boot. vgimportclone renames the VG and rewrites the PV UUIDs in one atomic step, eliminating the collision.

06 Activate the imported VG and rename the LV ACTIVATE ▶
# Activate the renamed VG vgchange -ay newvg2 # Rename the imported root LV to something descriptive lvrename /dev/newvg2/root /dev/newvg2/recovroot # Sanity-check VG and LV state before proceeding vgs lvs

Naming: newvg2 and recovroot are placeholders. Use anything that does not conflict with existing names on the host. Both become part of the device-mapper path: /dev/mapper/newvg2-recovroot.

07 Repair the filesystem and regenerate its UUID REPAIR ▶
XFS — RHEL / CentOS / Alma / Rocky / OL xfs_repair -L /dev/mapper/newvg2-recovroot xfs_admin -U generate /dev/mapper/newvg2-recovroot ext4 — Debian / Ubuntu e2fsck -y /dev/mapper/newvg2-recovroot tune2fs -U random /dev/mapper/newvg2-recovroot

xfs_repair -L zeroes the log. Any in-flight transactions on the recovered FS are discarded. Acceptable for a recovery scenario where the source system is gone, but never run on a healthy mounted filesystem.

UUID regen is not optional. If the recovered FS shares a UUID with the host's own root, mounts, /etc/fstab entries, and initramfs lookups all become ambiguous — this is the single most common cause of "host boots into the wrong root" after a recovery.

08 Mount and verify VERIFY ▶
# Create a temporary mount point mkdir -p /mnt/tmp # Mount the recovered LV mount /dev/newvg2/recovroot /mnt/tmp # Confirm it is mounted and accessible df -h ls /mnt/tmp

Recovery is now complete. The original filesystem tree is browseable under /mnt/tmp. Once you've extracted what you need, umount /mnt/tmp and detach the disk from the VM in vSphere.

⚠ Key Notes
  • Always re-read disk identity from lsblk. The /dev/sdX letter assigned to a freshly-attached disk is order-of-discovery — what was sdb on the source system may be sdf on the recovery host.
  • +100%FREE vs -L 20G. The first consumes all VG free space. The second sets absolute size if you forget the leading + — and will shrink an existing LV without warning.
  • XFS cannot shrink. Once xfs_growfs has expanded a filesystem there is no supported way to make it smaller. Plan growth deliberately.
  • vgimportclone is the safe path for recovery. Activating a foreign VG by hand risks colliding with the host's own VG names and UUIDs. The clone tool exists exactly to handle this.
  • UUID regen is mandatory after recovery. Skip it and the next reboot may pick the recovered FS as root. Run xfs_admin -U generate or tune2fs -U random before mounting.
  • xfs_repair -L destroys the log. Acceptable for recovery, never on a healthy production FS — fix mount issues first, log replay only as last resort.
  • If lsblk doesn't show the new disk after echo "- - -", the SCSI host enumeration may need a reboot — but check virt-layer hot-add is enabled first; rebooting a VM you're trying to recover into is a bad order of operations.
