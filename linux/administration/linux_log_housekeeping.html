Linux Log Housekeeping

Reactive disk-space recovery on Linux servers (truncate, compress, clear caches) and long-term logrotate policy to prevent recurrence.

Why two parts Reactive vs preventative

Part 1 is the response when a Linux server is full now: truncate, compress, delete, clear caches. It restores capacity but does nothing to stop recurrence.

Part 2 is the fix: drop a config file under /etc/logrotate.d/ and the daily cron hook handles rotation, compression, and retention going forward — no service restart, no manual intervention.

If you only ever do Part 1, you'll be back here in a fortnight. Always close the loop with Part 2 once the immediate fire is out.

Part 1 Free space now
01 Triage — find what's actually using the space DISCOVER ▶
# Filesystem-level view — confirm which mount is full df -h # Top 20 directories under /var/log by size du -shx /var/log/* | sort -h | tail -20 # Top files under /var across the whole subtree du -ahx /var 2>/dev/null | sort -h | tail -20

Tip: the -x flag stops du crossing into other filesystems — useful when / and /var sit on different mounts and you want to attribute usage correctly.

02 Truncate active system logs CLEAR ▶
# Truncate to zero bytes — preserves inode and any open file handles truncate -s 0 /var/log/syslog /var/log/messages # Alternative — for logs not actively being written to echo > /var/log/somelogname.log # Clear lastlog (read first if needed: lastlog | less) cp /dev/null /var/log/lastlog

Use truncate, not rm on logs the kernel or a daemon has open. Removing the file leaves the process writing to a now-orphaned inode — the bytes still exist, the file is "gone" from ls, and df won't recover until the holder is restarted. Truncation preserves the inode and frees space immediately.

03 Clear application logs CLEAR ▶
# TSM app logs — safe to zero (not delete) on systems we manage rm -f /usr/tivoli/tsm/client/oracle/bin64/logs/* # Other historical logs no longer needed rm -f /var/log/oldunusedlog.log

Always confirm a log can be removed before deleting it. Application logs may be required for audit, support cases, or compliance — when in doubt, archive off the system first (rsync/scp to a retention host) and remove only after the copy is verified.

04 Compress logs being kept for reference COMPRESS ▶
Compress — replaces .log with .log.gz gzip /var/log/somelogname.log Read without decompressing zcat /var/log/somelogname.log.gz Decompress — removes the .gz gunzip /var/log/somelogname.log.gz Decompress — keep both files gzip -d -c /var/log/somelogname.log.gz > /var/log/somelogname.log

Use gzip -9 for maximum compression. Best applied to old logs retained for audit — compressed files cannot be appended to or monitored without decompression, so don't compress logs that anything is still writing to.

05 Clean package cache and old kernels CLEAR ▶
Package cache # Ubuntu / Debian apt-get clean # RHEL / Rocky / Alma / OL dnf clean all Old kernels — keep at least one backup # Ubuntu / Debian apt autoremove --purge # RHEL / Rocky / Alma / OL package-cleanup --oldkernels --count=1 Other quick wins # Inspect /tmp and clear what's safe to remove ls -la /tmp

Always keep one backup kernel. If the running kernel develops a problem after a future update, that backup is what you boot into to recover. --count=1 on RHEL family and autoremove --purge on Debian both respect this by default — but verify with uname -r and ls /boot afterwards.

Part 2 Configure logrotate
06 Create the logrotate config drop-in CREATE ▶
# Confirm logrotate is present and the daily cron hook is in place which logrotate ls -la /etc/cron.daily/logrotate # Drop-in config — picked up automatically by the daily cron hook vi /etc/logrotate.d/somelogfilename

No service restart needed. Files in /etc/logrotate.d/ are read every time logrotate runs. The default cron hook fires it daily — for size-driven rotation you can also invoke logrotate from a more frequent cron entry, but for typical log volumes daily is sufficient.

07 Config contents — annotated template TEMPLATE ▶
# Author: Your Name // Change Number: CRXXXXXX # Service Express 2024 # ----------------------------------------------------------------------------- /path/to/your/log/file.log { missingok # Skip rotation if the log file is missing. notifempty # Don't rotate if the log is empty. weekly # Cadence: daily / weekly / monthly / yearly. rotate 4 # Keep the last 4 archives, delete older. compress # Compress rotated archives to save space. create 0644 root root # Create new log file post-rotation. # size 100M # Rotate on size rather than schedule (commented). maxsize 100M # Force rotation if log hits this size before schedule. postrotate > /path/to/your/log/file.log # Truncate after rotation. endscript }

Replace /path/to/your/log/file.log with the real log path before saving. The truncate (> /path...) inside postrotate is only needed when the writing process holds the file open and won't reopen on rotation — for most well-behaved daemons, create on its own is sufficient. Use copytruncate instead of create+postrotate when in doubt.

08 Validate, then force a real run VERIFY ▶
Dry run — checks syntax and reports what would happen sudo logrotate -d /etc/logrotate.d/somelogfilename Force a real run sudo logrotate -f /etc/logrotate.d/somelogfilename Verify # Confirm rotated archives appeared ls -la /var/log/somelogname* # Confirm the state file recorded the rotation grep somelogname /var/lib/logrotate/logrotate.status

Always dry-run first. -d shows the actions logrotate would take without making changes — read the output before forcing. -f bypasses the schedule check, so use it only after the dry run looks correct.

⚠ Key Notes
  • Truncate, don't rm, on open logs. A removed file held open by a daemon does not free space until the daemon closes the handle — typically a restart. truncate -s 0 frees space immediately and preserves the inode.
  • Always close Part 1 with Part 2. Reactive cleanup buys time; logrotate prevents recurrence. Don't leave the loop open.
  • copytruncate vs create. Use create for daemons that reopen log handles on rotation (most syslog implementations, journald-shipped logs). Use copytruncate for application scripts that append with >> and never reopen.
  • delaycompress avoids gzipping a file in flight. Skips compression on the most recent rotation. Worth adding to any policy where the writing process may briefly continue writing post-rotation.
  • Always dry-run before forcing. logrotate -d is harmless; logrotate -f rotates regardless of schedule. Read the dry-run output first, every time.
  • Keep at least one backup kernel when running package-cleanup --oldkernels or apt autoremove --purge. The default --count=1 respects this — confirm with ls /boot after.
  • If df shows full but du doesn't account for it, look for deleted-but-open files: lsof | grep deleted. Restarting the holder reclaims the space.
  • Don't compress active logs. Gzipped files cannot be appended to or tailed without decompression. Compression is for retention, not in-flight rotation.
