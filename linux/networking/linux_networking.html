Reference for everyday Linux network administration on RHEL 8/9 (and derivatives) using NetworkManager + firewalld. Covers identifying NICs, assigning addresses, creating new connections, persistent routes, DNS, and host-firewall rules. On RHEL 9 the legacy network-scripts package is gone — do not hand-edit ifcfg-* files while NetworkManager is running; everything below treats NetworkManager as the source of truth.

Identify What you have
01 Inventory NICs, addresses, drivers, link state REFERENCE ▶
# Link-layer view: NIC names, MAC, link state ip link # All addresses on all interfaces ip -br addr # NetworkManager view (the source of truth on modern RHEL) nmcli device status nmcli connection show # PCI / hardware mapping lspci | grep -i ether lshw -class network -short # Driver / firmware ethtool -i eth0 ethtool eth0 # link speed / duplex

Naming on RHEL 8/9 follows the predictable scheme — ens192, eno1, enp0s3 — substitute your actual interface name everywhere below. nmcli device status and nmcli connection show are the two commands you'll keep open in another pane while making changes.

NetworkManager Fundamentals
02 Devices vs connections, where profiles live, three edit modes CONCEPT ▶

NetworkManager separates two concepts: a device is the physical or virtual NIC (eth0, ens192, bond0); a connection is a saved profile that, when "up", configures a device. A device can have many saved connections — only one is active at a time. Profiles live in /etc/NetworkManager/system-connections/*.nmconnection.

systemctl status NetworkManager nmcli general status # List connections (saved profiles) and devices (hardware) nmcli connection show nmcli device status # Profiles on disk ls -l /etc/NetworkManager/system-connections/

Three ways to edit a connection: (1) nmcli — scriptable, idempotent, what you use 90% of the time; (2) nmtui — curses TUI, good for one-off changes on a console; (3) edit the .nmconnection file directly, then nmcli connection reload.

Do not mix NetworkManager with manual ifcfg-* edits on RHEL 9. The legacy network-scripts package is gone in RHEL 9 — there is no fall-back parser. Treat the .nmconnection profile as the only on-disk truth.

Configure DHCP interface
03 Add a new ethernet connection on DHCP CREATE ▶
# Simplest case — new NIC plugged in, want it on DHCP nmcli connection add \ type ethernet \ con-name eth1 \ ifname eth1 \ ipv4.method auto \ ipv6.method auto \ autoconnect yes nmcli connection up eth1 # Verify ip -br addr show eth1 nmcli -p connection show eth1 | grep -E "ipv4|state"

A new ethernet connection with DHCP gets you a working interface in three moves: add, up, verify. autoconnect yes means the connection will come up automatically on boot whenever the underlying device is present.

Configure Static IPv4
04 Add a static IPv4 connection (address, gateway, DNS) CREATE ▶
nmcli connection add \ type ethernet \ con-name ens224 \ ifname ens224 \ ipv4.method manual \ ipv4.addresses 10.20.30.40/24 \ ipv4.gateway 10.20.30.1 \ ipv4.dns "10.20.30.5 10.20.30.6" \ ipv4.dns-search "corp.example.com" \ ipv6.method disabled \ autoconnect yes nmcli connection up ens224

ipv4.addresses accepts a comma-separated list to assign multiple IPs to one interface — useful for service migrations and floating addresses.

Only one connection per default-route table should set ipv4.gateway — usually the management NIC. Two connections each declaring a default gateway will produce two default via routes and unstable egress (see Common failures).

Set ipv6.method disabled if your shop is IPv4-only; otherwise leave defaults so SLAAC / DHCPv6 still work.

Modify Change addresses, DNS, etc.
05 Edit addresses, append/remove with +/-, switch DHCP → static MODIFY ▶
# Change IP / gateway / DNS nmcli connection modify ens192 ipv4.addresses 10.0.0.50/24 nmcli connection modify ens192 ipv4.gateway 10.0.0.1 nmcli connection modify ens192 ipv4.dns "10.0.0.5" # Append a second IP nmcli connection modify ens192 +ipv4.addresses 10.0.0.51/24 # Remove an IP nmcli connection modify ens192 -ipv4.addresses 10.0.0.51/24 # Switch DHCP → static nmcli connection modify ens192 ipv4.method manual # Apply changes (re-up the connection) nmcli connection down ens192 && nmcli connection up ens192

Changes are written to /etc/NetworkManager/system-connections/<name>.nmconnection immediately — the up/down cycle is what makes the kernel reflect them. Use the + / - prefix to append or remove individual entries from list-typed properties (addresses, routes, DNS).

Tooling nmtui
06 Console-only TUI for editing connections and hostname REFERENCE ▶
nmtui

Edit a connection → pick connection → tab to fields → fill IPv4 / IPv6 / Routes → OK. Activate a connection → toggles up/down. Set system hostname → updates /etc/hostname and notifies systemd-hostnamed.

nmtui writes the same .nmconnection files as nmcli — safe to alternate between the two. Useful when you're at a serial console and don't want to type out a full nmcli incantation.

Routing Persistent routes
07 Add a single static route to a connection ADD ▶
# Route 192.168.50.0/24 via 10.0.0.254, attached to ens192 nmcli connection modify ens192 +ipv4.routes "192.168.50.0/24 10.0.0.254" # Reload to apply nmcli connection up ens192 # Verify ip route ip route show 192.168.50.0/24

A single +ipv4.routes entry is the cleanest way to attach a persistent next-hop to one connection. The route is bound to the connection profile — when the connection comes up, the kernel installs the route; when it goes down, the route disappears. This is exactly the persistence you want.

08 Multiple routes, metric, table, delete — and what's on disk REFERENCE ▶
# Route with explicit metric nmcli connection modify ens192 +ipv4.routes "192.168.50.0/24 10.0.0.254 100" # Form: "<dest>/<prefix> <gateway> [<metric>] [table=<id>]" # Replace the entire route list (no leading +) nmcli connection modify ens192 ipv4.routes \ "192.168.50.0/24 10.0.0.254, 192.168.60.0/24 10.0.0.254, 172.16.0.0/16 10.0.0.253" # Delete a single route nmcli connection modify ens192 -ipv4.routes "192.168.50.0/24 10.0.0.254" On disk — /etc/NetworkManager/system-connections/ens192.nmconnection [ipv4] method=manual addresses1=10.0.0.50/24,10.0.0.1 routes1=192.168.50.0/24,10.0.0.254 dns=10.0.0.5;

Bare ipv4.routes replaces the list — use +ipv4.routes to append and -ipv4.routes to remove a single entry. Forgetting the + when you meant to append will silently drop every other static route on the connection.

DNS Resolution
09 Per-connection DNS, ignore-auto-dns, verification REFERENCE ▶
# Set DNS servers and search domains on a connection nmcli connection modify ens192 ipv4.dns "10.0.0.5 10.0.0.6" nmcli connection modify ens192 ipv4.dns-search "corp.example.com lab.example.com" # Override DNS from this connection only — ignore DHCP-supplied DNS nmcli connection modify ens192 ipv4.ignore-auto-dns yes # Re-up to apply nmcli connection up ens192 Verify resolvectl status # if systemd-resolved is enabled cat /etc/resolv.conf getent hosts mail.corp.example.com dig +short corp.example.com

Do not edit /etc/resolv.conf by hand while NetworkManager is running. On RHEL it's typically a managed file (or a symlink to a stub maintained by systemd-resolved) — manual edits are clobbered the next time a connection comes up. Set DNS via nmcli on the connection profile instead.

Bonding LACP / active-backup
10 Two NICs into bond0 with two enslaved ports SWITCH-SIDE ▶
# Create the bond nmcli connection add type bond \ con-name bond0 ifname bond0 \ mode 802.3ad miimon 100 \ ipv4.method manual \ ipv4.addresses 10.0.0.50/24 \ ipv4.gateway 10.0.0.1 \ ipv4.dns 10.0.0.5 # Enslave the two physical NICs nmcli connection add type ethernet slave-type bond \ con-name bond0-port1 ifname ens192 master bond0 nmcli connection add type ethernet slave-type bond \ con-name bond0-port2 ifname ens224 master bond0 # Bring it up (NM handles slave→master ordering automatically) nmcli connection up bond0 Verify cat /proc/net/bonding/bond0 ip -br link

Common modes: active-backup, 802.3ad (LACP), balance-tlb, balance-alb. Match what the switch supports.

Switch-side configuration must match the bond mode. Running the bond as 802.3ad while the switch port is configured for static aggregation (or vice versa) gives you a bond that "comes up" on the host but never passes traffic — confirm with cat /proc/net/bonding/bond0 and look for "Currently Active Slave" / LACP partner state.

VLAN Tagging
11 Tag VLAN 200 on ens192 as a child interface REFERENCE ▶
nmcli connection add type vlan \ con-name ens192.200 \ ifname ens192.200 \ dev ens192 \ id 200 \ ipv4.method manual \ ipv4.addresses 10.200.0.10/24 nmcli connection up ens192.200

The trunk side of the switch port must be configured to allow VLAN 200, or the child interface will be up but isolated. The naming convention <parent>.<vlan-id> isn't required, but it's the standard you'll find on every Linux host.

firewalld State and zones
12 Inspect state, default zone, active zones, current rules REFERENCE ▶

firewalld is a stateful zone-based wrapper around nftables (or iptables on older systems). The default zone on RHEL is public. Each interface lives in exactly one zone; rules are evaluated per-zone.

systemctl status firewalld firewall-cmd --state firewall-cmd --get-default-zone firewall-cmd --get-active-zones # Rules in the default zone firewall-cmd --list-all # Rules in a specific zone firewall-cmd --zone=public --list-all

Service definitions live in /usr/lib/firewalld/services/*.xml (defaults, do not edit) and /etc/firewalld/services/*.xml (your overrides). Use firewall-cmd --info-service=<name> to see what a named service expands to.

firewalld Open a port / allow a service
13 Add a port or named service permanently and reload ADD ▶
# Open a port — write to config, then reload to make active firewall-cmd --permanent --zone=public --add-port=8080/tcp firewall-cmd --reload # Verify firewall-cmd --zone=public --list-ports Allow a named service firewall-cmd --permanent --add-service=https firewall-cmd --reload firewall-cmd --list-services # What does that service expand to? firewall-cmd --info-service=https

Forgetting --permanent + --reload is the #1 firewalld footgun. Without --permanent the change is runtime only and disappears on reload or reboot. With --permanent alone the change is on disk but not yet active — you still need --reload for the live ruleset to pick it up. Pair them every time, or use firewall-cmd --runtime-to-permanent after testing live.

firewalld Restrict to source / change zone
14 Rich rule for source-restriction; move a NIC to a different zone RICH RULE ▶
# Allow SSH only from 10.0.0.0/24 firewall-cmd --permanent --zone=public --add-rich-rule=' rule family="ipv4" source address="10.0.0.0/24" port port="22" protocol="tcp" accept' firewall-cmd --reload firewall-cmd --list-rich-rules Move a NIC to a more-trusted zone # Move ens224 from 'public' to 'internal' firewall-cmd --permanent --zone=internal --change-interface=ens224 firewall-cmd --reload firewall-cmd --get-active-zones

Rich rules give you the full nftables-flavoured grammar — source/destination address, port + protocol, action, optional logging — without needing to touch nftables directly. Keep them readable; multi-line quoted form is supported and recommended.

An interface lives in exactly one zone at a time. --change-interface is the right verb when you want a NIC to be evaluated against a different rule set (e.g. trusted management LAN moves to internal; DMZ-facing NIC stays on public).

firewalld Remove rules
15 Remove a port / service, save runtime ruleset to permanent REMOVE ▶
# Remove a port firewall-cmd --permanent --zone=public --remove-port=8080/tcp # Remove a named service firewall-cmd --permanent --remove-service=cockpit # Apply firewall-cmd --reload Save what you tested live to permanent # If you've been adding rules WITHOUT --permanent and want to keep them firewall-cmd --runtime-to-permanent

Removing a service can lock you out of an in-flight session. Always confirm you have an alternate route in (console, OOB, secondary NIC in a different zone) before --remove-service=ssh or --remove-port=22/tcp on the management interface.

--runtime-to-permanent snapshots the live ruleset to disk — handy when iterating without --permanent, then committing once you're happy.

Verification Toolkit
16 Addresses, routes, ARP, reachability, path, sockets, DNS, NM log REFERENCE ▶
Addresses + routes ip -br addr ip -br link ip route ip -6 route ARP / neighbor table ip neigh Reachability ping -c 3 10.0.0.1 ping -c 3 8.8.8.8 Path tracepath 8.8.8.8 mtr -rwc 20 8.8.8.8 Listening sockets ss -tlnp ss -tunp DNS resolvectl status dig +short corp.example.com getent hosts host.corp.example.com NetworkManager event log journalctl -u NetworkManager --since "10 min ago"

Work top-to-bottom when something is broken: confirm the address is on the interface, confirm a route exists toward the destination, confirm the host is reachable, confirm the path looks sensible, confirm the service is listening, confirm DNS resolves. Skipping a layer wastes time.

⚠ Key Notes
  • NetworkManager is the source of truth on RHEL 8/9 — manage connections via nmcli / nmtui, not legacy ifcfg-* scripts (the network-scripts package is gone in RHEL 9).
  • Only one connection per default-route table should set ipv4.gateway — two default routes produce flapping egress.
  • Persistent static routes belong in the connection profile (+ipv4.routes), not in /etc/sysconfig/network-scripts and not via one-shot ip route add — those don't survive a reboot.
  • firewalld changes need --permanent and --reload to be both saved and active. Use one without the other and you'll be debugging a phantom rule.
  • firewall-cmd --runtime-to-permanent saves the current live ruleset to disk — useful after iterating with runtime-only changes.
  • Set ipv4.ignore-auto-dns yes on a static connection to prevent DHCP-supplied DNS from overriding your configured resolvers.
  • Don't hand-edit /etc/resolv.conf while NetworkManager is running — it's a managed file and your changes will be clobbered.
  • Switch-side LACP / aggregation mode must match the bond mode exactly. A "bond up, no traffic" symptom is almost always switch-side mismatch — confirm via cat /proc/net/bonding/bond0.
