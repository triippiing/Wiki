User administration on AIX maps closely onto its Linux equivalent — the lifecycle and concepts are identical, but the binaries differ. Edits are committed atomically through the high-level commands below; do not hand-edit /etc/passwd or /etc/group on AIX, as the security ODM and shadow files (/etc/security/passwd, /etc/security/user) must be kept in sync.

Reference Common mkuser flags
01 Flag reference for mkuser REFERENCE ▶
FlagPurpose
-aAssigns an attribute to the user (e.g. -a home=/home/trip) — used to set the home directory path and other stanza attributes.
-BWhen run with mkuser, creates the home directory specified via the -a home= attribute. Without it, the home path is recorded but the directory is not created on disk.
-cAdds a comment / GECOS-style description for the user.
-hSets the home directory for the user (alternate form, used directly rather than via -a home=).
-gSpecifies the user's primary group (GID).
-GSpecifies additional groups (comma-separated) the user belongs to.
-pSets the user's password. Rarely used directly — prefer passwd <user> after creation so the password is not stored in shell history.

Multiple -a attributes can be supplied in the same mkuser invocation — for example mkuser -a home=/home/trip shell=/usr/bin/bash gecos="Trip Doe" trip. All stanza attributes from /etc/security/user are settable via -a.

User Creation
02 Create user "trip" with home directory /home/trip CREATE ▶
# Create the account and build the home directory in a single call mkuser trip -a home=/home/trip -B # Set the initial password (interactive — will prompt for new password) passwd trip # Force password change at next login pwdadm -f ADMCHG trip # Verify the account stanza and home directory lsuser trip ls -ld /home/trip

The -B flag is what actually creates /home/trip on disk; without it the path is registered in /etc/passwd but the directory does not exist, and the user's first login will land in /. Ownership and the skeleton dotfiles (.profile, etc.) are populated automatically when -B is used.

User Removal
03 Remove user "trip" — account and home directory DESTRUCTIVE ▶
# Confirm the user exists and inspect home directory contents first lsuser trip ls -la /home/trip # Remove the user — this also removes the home directory rmuser trip # If you want to keep the home directory, preserve it before rmuser tar -cvf /backup/trip_home.tar /home/trip rmuser trip

rmuser is destructive. The home directory is removed along with the account. For staff who have left the organisation, archive any contents that may be required (mailboxes, shell history, scripts, .ssh keys for shared automation) before issuing rmuser. There is no built-in recovery.

If an account is being deactivated rather than deleted (audit retention, pending handover), prefer chuser account_locked=true trip instead — this keeps the home directory and audit trail intact while preventing login.

Group Creation
04 Create group "Service_Express" CREATE ▶
# Create the group mkgroup Service_Express # Confirm the group was created and inspect its current member list lsgroup Service_Express # Optionally set a specific GID at creation time mkgroup id=5050 Service_Express

By default mkgroup picks the next available GID. Specifying id= is useful when groups must align with NIM masters or NFS-shared filesystems on other hosts. Group names are case-sensitive on AIX.

Group Membership Management
05 Add user "trip" to "Service_Express" — preserving existing groups REPLACES LIST ▶
# Inspect trip's current group membership FIRST lsuser -a groups trip grep trip /etc/group # chuser groups= REPLACES the entire list — must include existing groups # Example: trip is currently in "admins" and we want to add "Service_Express" chuser groups=admins,Service_Express trip # Verify the new membership lsuser -a groups trip

chuser groups= replaces the full list, it does not append. Always read lsuser -a groups <user> first and supply the complete comma-separated list. Forgetting an existing group will silently remove the user from it — most commonly seen when an admin loses their system or security group membership.

06 Remove user "trip" from "admins" group REPLACES LIST ▶
# Read the current list — assume trip is in admins,Service_Express lsuser -a groups trip # Re-issue chuser with admins OMITTED from the new list chuser groups=Service_Express trip # Verify removal lsuser -a groups trip grep trip /etc/group

There is no separate "remove from group" command on AIX — removal is achieved by listing the desired membership and omitting the unwanted group. This is the same mechanism as adding a group, just with a different target list.

Group Removal
07 Remove group "admins" DESTRUCTIVE ▶
# Inspect the group's current membership BEFORE removal lsgroup -a users admins # Confirm no user has this group as their PRIMARY group lsuser -a pgrp ALL | grep "pgrp=admins" # Remove the group rmgroup admins # Verify removal — should report "does not exist" lsgroup admins

rmgroup also strips the group from every member's secondary group list. Confirm membership with lsgroup -a users <group> first, and ensure no user has the group as their primary group (pgrp) — rmgroup will fail in that case, and the user must have their pgrp reassigned via chuser pgrp= first. Only remove groups you are certain are safe to remove.

⚠ Key Notes
  • Do not hand-edit /etc/passwd, /etc/group, or the files under /etc/security/. Use the mkuser/chuser/rmuser family — they keep the security ODM and shadow files in sync.
  • chuser groups= replaces the user's secondary group list — it does not append. Always read the existing list first, edit it, then write it back.
  • rmuser deletes the user's home directory along with the account. Archive anything required (mailboxes, .ssh, shell history) before issuing the command.
  • For staff awaiting handover or audit retention, prefer chuser account_locked=true <user> over rmuser.
  • rmgroup removes the group from every member's secondary list. Run lsgroup -a users <group> first; rmgroup will fail if any user has it as their primary group (pgrp).
  • Group and user names are case-sensitive on AIX — Service_Express and service_express are different groups.
  • When aligning UIDs/GIDs across NIM masters or NFS clients, supply id=N at creation time rather than letting AIX auto-pick.
