First-pass triage when an AIX system is reported slow. The goal is to classify the bottleneck — CPU, memory, disk, network, or lock contention — within roughly five minutes using a top-down sequence of topas → vmstat → branch tool, then drill into the responsible subsystem. Works for both live "it's slow right now" calls and post-mortem analysis from nmon recordings.

Step 0 Frame the symptom
01 Four framing questions before any tool REFERENCE ▶
QuestionWhy it matters
Now or earlier?Slow right now calls for live tools (topas, vmstat, iostat). Slow earlier needs historical data — pull the nmon -f recording from /var/perf.
Whole system, app, or user?Narrows scope from system-wide tuning down to per-process drilling. A single user complaining usually means an app or per-process issue, not a host-level bottleneck.
When did it start?Correlate with recent change — TL upgrade, new application rollout, LPAR resize, SAN event, errpt entries. Recent change is the leading suspect.
Slow or hung?Hung ≠ slow. Hung often points at resource starvation, kernel deadlock, or storage stall — jump to the Hang branch at the end and skip the regular flow.

Spending 30 seconds on these questions before opening any tool prevents the most common triage mistake — diving into topas on a host that is actually hung, not slow, where the data you need is in the dump device, not the live console.

Step 1 Snapshot with topas
02 One-shot snapshot — read User%, Kern%, Wait%, Idle% REFERENCE ▶
# 5-second refresh, 2 iterations — quick top-of-screen read topas -d 5 -i 2 # Top-of-screen example output # CPU User% Kern% Wait% Idle% # ALL 82.0 12.0 1.0 5.0

Read the top three lines of topas first. The CPU breakdown alone routes you to the correct branch in under five seconds.

ReadingFirst suspicion
User% > 70Application-level CPU burn — go to the CPU branch, drill with tprof and per-process tools.
Kern% > 30Kernel work — syscalls, interrupts, lock contention. Use the kernel-CPU sub-section in the CPU branch.
Wait% > 20I/O wait — go to the Disk/Network branch. Confirm with vmstat column b.
All low + slowCPU isn't the bottleneck. Look at Memory (paging) or Lock contention.

Also scan the lower panels of topas for corroborating signals: Memory — Comp% > 90 indicates memory pressure; Paging — non-zero Faults or pgspaceIn/Out means active paging; Disk Busy% — any disk consistently above 60% is a hot spot; Network — KBPS approaching link cap means network bound.

Step 2 Confirm with vmstat
03 Cross-check the topas read with vmstat columns REFERENCE ▶
# 2-second interval, 10 samples — discard the first line (boot averages) vmstat 2 10
ColumnMeaning & bad sign
rRun queue length. Sustained r > CPU count = CPU saturation.
bBlocked on I/O. b > 0 sustained = I/O bound.
avmActive virtual memory pages. Climbing = memory growth / leak.
freFree pages. Approaching minfree = memory pressure.
pi/poPaging in/out. Any sustained non-zero value is bad — actively paging to disk.
fr/srPage replacement scan rate. High values mean the VMM is working hard to free pages.
usUser CPU% — mirrors topas User%.
sySystem CPU% — mirrors topas Kern%.
idIdle CPU%.
waI/O wait CPU% — mirrors topas Wait%.

Heuristic shorthand: b > 0 with high wa% = disk wait. r >> CPUs with high us% = CPU saturation. pi/po non-zero = paging — find the leak before adding paging space.

Branch CPU
04 CPU branch — User% high, plus kernel-CPU sub-section DRILL-DOWN ▶
USER% HIGH — per-process / per-thread# Live per-process CPU view topas -P # Top CPU consumers (one-shot) ps -eo pcpu,pid,user,args | sort -k1 -rn | head # Per-thread breakdown for a specific PID ps -mp <PID> -o THREAD # Sample where the CPU is going (user + kernel) for 30 seconds tprof -uskez -x sleep 30 more __prof.all KERN% HIGH — kernel CPU drilling# Kernel-only sampling tprof -ksez -x sleep 30 # Trace syscalls + dispatches for 5 seconds trace -aj 100,200 -o /tmp/trc; sleep 5; trcstop trcrpt -O exec=on,pid=on /tmp/trc | head -50 # Kernel lock activity lockstat -l 30

tprof tells you the top CPU consumers per process and per function, and whether time is in user code, libc, or kernel. If tprof shows excessive lock_*, page faults, or interrupt handlers in the kernel column, you have kernel CPU spent on contention rather than real work — pivot to lockstat.

Branch Memory
05 Memory branch — fre low, pi/po non-zero, or comp% high DRILL-DOWN ▶
# One-line summary of every memory tunable + counter vmstat -v # Global memory usage breakdown svmon -G # Top 10 processes by memory consumption svmon -P -O summary=basic -t 10 # Paging space layout and use lsps -a # Find leak candidates — sort processes by virtual size ps -eo pid,user,vsz,rss,args --sort -vsz | head

Read vmstat -v for: numperm percentage (file cache portion), pending disk I/Os blocked with no pbuf (pbuf shortage), paging space allocs / I/Os (active paging), and client filesystem I/Os blocked with no fsbuf (fsbuf shortage). Healthy when numperm sits between minperm% and maxperm%; trouble when numclient is near maxclient% with active paging.

Computational vs file pages. AIX distinguishes computational pages (process working memory — paged to disk under pressure) from file pages (filesystem cache — discarded under pressure). Paging out computational memory is the painful kind. If pi/po are non-zero, find the leak before adding paging space — extra paging space buys time, not a fix.

Branch Disk / I/O
06 Disk branch — Wait% high or vmstat b non-zero CAUTION ▶
PER-DISK / ADAPTER / LV BREAKDOWN# Per-disk activity with service time iostat -DRT 2 5 # Adapter-level summary iostat -aT 2 5 # Per-LV / VG breakdown iostat -ARTl 2 5 FILE-LEVEL ATTRIBUTION# Trace which process is hitting which file filemon -O all -o /tmp/fm.out; sleep 30; trcstop more /tmp/fm.out MPIO / PATH HEALTH# Full path inventory lspath # Per-disk path detail lspath -l hdisk5 -F "name parent path_id status" # Path failures (Storage Manager / SDDPCM error ID) errpt -j FE9C3F69 | head

Read iostat -DRT for: % tm_act (disk busy — sustained > 60 on one disk = hot spot), avg serv (service time in ms — > 20 ms on SAN suggests path or backend), and avwait (queue wait — climbing means device queue is full).

One disk vs. all disks. High service time on a single disk points at a bad path, full LUN queue, or a backend issue specific to that LUN. High service time on all disks at once almost always points at SAN fabric, the HBA, or VIOS — escalate to storage and stop drilling per-disk.

Branch Network
07 Network branch — interface errors, mbuf failures, MTU mismatch DRILL-DOWN ▶
# Per-interface stats — top of output is the headline entstat -d en0 | head -60 netstat -v en0 | head -40 # Buffer / mbuf health — look for "failures" lines netstat -m | head -20 # Connection state distribution — TIME_WAIT pile-up, CLOSE_WAIT leaks, etc. netstat -an | awk '{print $6}' | sort | uniq -c # Network tunables — buffer ceilings no -a | grep -E "tcp_sendspace|tcp_recvspace|sb_max"

Watch for: non-zero Transmit Errors or Receive Errors, non-zero S/W Transmit Queue Overflow, any failures line in netstat -m, and MTU mismatch versus the upstream switch or SEA. netstat -m failures are the classic signal of buffer exhaustion under load.

Branch Lock contention
08 Lock contention — system slow but CPU/disk/mem all idle DRILL-DOWN ▶
# Kernel lock activity — 30-second sample lockstat -l 30 # Process state — many processes blocked ('A' = active, $4 = wait flag) ps -ekf | awk '$3=="A" && $4>0' | head # Trace lock acquisition events for 10 seconds trace -aj 1B0,1B1 -o /tmp/lk; sleep 10; trcstop trcrpt -O exec=on /tmp/lk

Lock contention is the diagnosis when the host is reported slow but every other branch comes up clean — CPU has idle, disk is quiet, memory is healthy, network is fine. Many processes will sit in wait state on a kernel resource. lockstat -l ranks the hot locks; the trace hooks 1B0/1B1 show the lock acquire/release path.

Recording nmon for historical & vendor data
09 nmon — record now, analyse later, baseline forever REFERENCE ▶
# Record: 30s samples × 240 = 2 hours, file output to /var/perf nmon -s 30 -c 240 -f -m /var/perf # Live interactive — then press the section key: # c = cpu m = memory d = disk n = network t = top procs nmon # Output filename: /var/perf/<host>_YYYYMMDD_HHMM.nmon # Open with nmon Analyser (Excel) or nmonchart for graphs

For continuous baseline, schedule via cron — keeps a rolling 24h window of system telemetry that becomes invaluable when a vendor or IBM Support asks "what was it doing yesterday at 14:00?".

# /var/spool/cron/crontabs/root — one nmon launch per hour, 30s samples × 120 = 1h 0 * * * * /usr/bin/nmon -s 30 -c 120 -f -m /var/perf -T
Decision Tree One-page routing
10 topas → branch routing diagram REFERENCE ▶
┌── User% > 70 ─────► tprof user, ps -mp <PID> -o THREAD CPU bound ──────┤ (us+sy idle low) └── Kern% > 30 ─────► tprof kernel, trace, lockstat ▲ │ topas ────┼──► Wait% > 20 ──► iostat -DRT ──► hot disk? ── yes ─► filemon, lspath │ │ │ └─ no ─► all disks slow ─► SAN/VIOS/HBA │ ├──► fre low / pi-po non-zero ─► svmon -G, vmstat -v ─► leak vs cache │ ├──► entstat errors / netstat -m fails ─► network branch │ └──► all idle but slow ─► lockstat -l ─► lock contention

Read top-down from topas: the dominant CPU mode (User%, Kern%, Wait%) routes you into the correct branch, and the side-branches catch the cases topas CPU summary doesn't surface (memory, network, locks).

Hang Branch Unresponsive but pingable
11 Hang branch — capture state for IBM Support CRITICAL ▶
# From console — kernel debugger snapshot (live) echo "stat" | kdb # Inspect dump device configuration sysdumpdev -L # Trigger a live system dump to the primary device sysdumpstart -p # After recovery — flush errpt for IBM errpt -a > /tmp/errpt.live

If the system is unresponsive over the network too, the console is the only option. Capture HMC reference codes (LED), the full errpt -a output once recovered, and a live or post-mortem dump for IBM Support. sysdumpstart -p will take the system down — confirm policy before issuing on a production LPAR.

⚠ Key Notes
  • Always start with topas before drilling — the User%/Kern%/Wait%/Idle% read at the top tells you which branch to take in five seconds.
  • vmstat column b > 0 sustained = I/O wait; cross-check with high wa% and head straight to iostat -DRT.
  • Sustained r > CPU count on vmstat means CPU saturation — the run queue is longer than the host can drain.
  • pi/po non-zero means active paging. Find the leak before adding paging space — extra paging is a workaround, not a fix.
  • % tm_act > 60 on a single hdisk = hot spot. The same number across all disks at once = SAN, HBA, or VIOS issue — escalate to storage rather than drilling per-disk.
  • Idle CPU but the host is reported slow = lock contention. Run lockstat -l 30 and trace hooks 1B0/1B1.
  • Capture an nmon recording (-s 30 -c 240 -f -m /var/perf) for IBM Support or the application vendor — it is the artefact every escalation will ask for.
  • Slow ≠ hung. If the host is unresponsive, skip the regular flow and go to the Hang branch — collect kdb stat, dump device config, and HMC LED codes.
