Runbook for establishing passwordless SSH from a management host (trips-testlab) to an IBM HMC (drhmc01) for unattended monitoring and automation. The HMC's restricted shell means ssh-copy-id won't work — use mkauthkeys instead.

Step 0 Prerequisites
00 Confirm reachability, credentials, and access REFERENCE ▶

Verify the following before commencing:

  • · Network reachability on TCP/22 from 192.168.131.135 to 192.168.131.150.
  • · hscroot password available — required once for the initial key upload.
  • · Root access on trips-testlab (or whichever local user will run the automation).
  • · An SCP/PuTTY session to the HMC for paste operations.
Step 1 Generate the SSH key pair
1.1 Create the ~/.ssh directory with strict permissions trips-testlab ▶

The ~/.ssh directory must exist with mode 700 before key generation, or ssh will refuse to use the keys.

# run as root on trips-testlab mkdir -p ~/.ssh chmod 700 ~/.ssh
1.2 Generate a dedicated RSA 4096 keypair, no passphrase trips-testlab ▶

A dedicated key isolated from any pre-existing keys. Empty passphrase is required for unattended cron execution — security is enforced by filesystem permissions.

ssh-keygen -t rsa -b 4096 \ -C "hmc_monitor@trips-testlab" \ -f ~/.ssh/hmc_monitor_rsa \ -N ""

Passphrase: when prompted, leave empty by pressing Enter twice. A passphrase will block unattended execution under cron.

Verify: ls -l ~/.ssh/hmc_monitor_rsa* — should show two files: private key (mode 600) and public key .pub (mode 644).

Step 2 Configure SSH client alias
2.1 Add a host alias to ~/.ssh/config trips-testlab ▶

Lets the script reference drhmc01 by its short alias and have SSH automatically apply the correct user, IP, key, and connection options.

cat >> ~/.ssh/config <<'EOF' Host drhmc01 HostName 192.168.131.150 User hscroot IdentityFile ~/.ssh/hmc_monitor_rsa IdentitiesOnly yes StrictHostKeyChecking accept-new ConnectTimeout 10 BatchMode yes EOF chmod 600 ~/.ssh/config

Why IdentitiesOnly yes: without it, SSH offers every key in the agent before falling back to the configured key. HMCs lock out hscroot after several failed attempts — we want only the designated key to be presented.

Step 3 Upload the public key to the HMC
3.1 Display the public key on the source host trips-testlab ▶

Print the public key so it can be copied. Select the entire output as a single line — no line breaks when transferred.

cat ~/.ssh/hmc_monitor_rsa.pub

Expected output format — a single very long line:

ssh-rsa AAAAB3NzaC1yc2EAAA...(very long string)... hmc_monitor@trips-testlab
3.2 SSH to the HMC interactively (last password-required step) trips-testlab ▶

Authenticate with the hscroot password one final time to register the key.

ssh hscroot@192.168.131.150

Do not use ssh-copy-id: the HMC's restricted shell does not support its mechanism for appending to authorized_keys. Use the HMC-native mkauthkeys instead (next step).

3.3 Register the public key with mkauthkeys drhmc01 ▶

From the HMC prompt, paste the full public key as the argument to mkauthkeys -a. Wrap in double quotes so the comma and trailing comment are preserved.

mkauthkeys -a "ssh-rsa AAAAB3Nza...full_key... hmc_monitor@trips-testlab"

Replace the example string above with the actual key copied in 3.1.

Verify: mkauthkeys --ls — your key should appear in the list. Inspect the trailing comment (hmc_monitor@trips-testlab) to confirm the correct key was added.

3.4 Exit the HMC session drhmc01 ▶
exit
Step 4 Verify passwordless authentication
4.1 Test connection using the configured alias trips-testlab ▶

This single command exercises the full chain: config alias → key selection → HMC authentication → remote command.

ssh drhmc01 lshmc -V

Expected: the HMC version block prints immediately, no password prompt. If the command returns the version, key authentication is fully functional.

Step 5 Troubleshooting
5.1 Verbose SSH auth trace + symptom-to-fix table DIAGNOSE ▶

For verbose output diagnosing the auth flow:

ssh -vvv drhmc01 lshmc -V 2>&1 | grep -E "(Offering|Authentication|denied|accept)"
SymptomLikely causeResolution
Prompts for password despite key being present Key not registered on HMC, or wrong key being offered mkauthkeys --ls on HMC; ensure key matches ~/.ssh/hmc_monitor_rsa.pub on collector
Permission denied (publickey) Permissions on private key, ~/.ssh, or ~/.ssh/config too open chmod 700 ~/.ssh; chmod 600 ~/.ssh/hmc_monitor_rsa ~/.ssh/config
Connection times out Firewall blocking TCP/22 or HMC unreachable nc -zv 192.168.131.150 22 from trips-testlab
Host key verification failed HMC was reinstalled / host key changed ssh-keygen -R 192.168.131.150; ssh-keygen -R drhmc01 then retry
Pasted key shows as multiple lines on HMC Terminal wrapping during paste mkauthkeys --rm the malformed entry, then re-paste with quotes ensuring a single line
mkauthkeys: command not found Not on the HMC restricted shell — shell escape or wrong host Confirm prompt shows hscroot@drhmc01; reconnect if necessary
Step 6 Rollback
6.1 Remove the public key from the HMC drhmc01 ▶

From an interactive hscroot session on the HMC:

mkauthkeys --ls mkauthkeys --rm "ssh-rsa AAAA...full_key... hmc_monitor@trips-testlab"
6.2 Remove the keypair on the collector trips-testlab ▶
rm -i ~/.ssh/hmc_monitor_rsa ~/.ssh/hmc_monitor_rsa.pub

Optionally remove the Host drhmc01 alias block from ~/.ssh/config if the host is being decommissioned.

▸ Watchouts
  • Empty passphrase is a deliberate choice for unattended cron — do not add one after the fact without also fixing the caller.
  • ssh-copy-id silently fails against HMCs — always use mkauthkeys.
  • If hscroot gets locked out from failed key attempts, the console (physical or remote KVM) is the only recovery path — bring up the account there first.
  • Rotate the keypair when the target host is retired, on a schedule, or after any suspected compromise of trips-testlab.
