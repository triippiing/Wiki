Purpose

Register a new VMware vCenter as a protection source on the Cohesity cluster. Registering the vCenter lets Cohesity discover the inventory — datacenters, clusters, hosts, datastores, VMs — and apply Protection Groups and policies against it.

Scope

Covers registration of a vCenter Server source only. Creating Protection Groups, policies, or recovery jobs against the newly registered source is handled by a separate runbook.

Phase 1 Prerequisites

Confirm all of the following before starting the registration.

Network / Firewall
  • TCP 443 open from the Cohesity cluster Node IPs and the Cluster VIP to the vCenter FQDN / IP.
  • DNS resolution working in both directions between Cohesity nodes and the vCenter.
  • NTP in sync between the Cohesity cluster and vCenter — skew > 5 minutes will cause SSO auth failures.
Service Account
  • A vCenter service account exists with at least the Cohesity-required vCenter role. Key privileges include Virtual machine.Snapshot management, Virtual machine.Provisioning, Datastore.Browse / Low-level file operations, Resource.Assign VM to resource pool, VirtualMachine.Interact.*, and Global.Disable methods / Enable methods — full list on Cohesity's VMware Requirements page.
  • Account is not subject to a password expiry policy that rotates without notice — or, if it is, the rotation process is integrated with Cohesity credential updates.
  • Credential is stored in the team password vault.
vCenter Health
  • No in-flight vCenter upgrade.
  • Adequate compute resource available for restore operations.
Cohesity Side
  • Cluster is on a supported version for the target vCenter version — cross-check the Cohesity Interop matrix.
  • Sufficient capacity headroom on the cluster for the projected ingest of the new vCenter.
  • Logged in to the Cohesity UI as a user with the Admin or Source Admin role.

Check the clock first. A time skew over 5 minutes between the cluster and vCenter causes SSO authentication to fail with a non-obvious error. Verify NTP sync on both sides before troubleshooting anything else during registration.

04 Configure registration options CRITICAL Cohesity UI ▶

Set the toggles exactly as below. Any toggle not listed must remain in its default OFF state.

OptionState

Auto Cancel — threshold: when Auto Cancel Backups if Datastore is running low on space is enabled you must set a threshold. Cohesity recommends setting it by percentage rather than GiB — percentage auto-scales as the datastore grows or shrinks and applies cleanly to VMs with disks spanning multiple datastores. Default recommendation: 10% free space minimum, adjusted to local datastore sizing.

05 Rationale for the three enabled options REFERENCE Cohesity UI ▶
OptionWhy it is enabled
Detect VM migration across vCenter If a VM is migrated (cross-vCenter vMotion, Storage vMotion across linked vCenters, etc.) Cohesity recognises it as the same VM and continues the existing backup chain rather than seeding a brand-new full backup. Saves ingest, capacity, and runtime.
Use VM BIOS UUID to track VMs BIOS UUID is more stable than MoRef or Instance UUID across VM re-registration, vCenter rebuild, and certain DR scenarios — improves chain continuity and avoids accidental reseeds. Caveat: if any VMs in this vCenter have duplicate BIOS UUIDs (sometimes seen after clone-without-regenerate) tracking becomes ambiguous — see Key Notes.
Auto Cancel Backups on low datastore space Prevents Cohesity-initiated VMware snapshots from consuming the last free space in a datastore and stunning / crashing production VMs. This is a safety-rail option and should be always on.
06 Complete registration CONFIG Cohesity UI ▶
  1. Click Register (or Complete, depending on cluster UI version).
  2. Wait for the source status to move from Registering → Online.

Discovery time: initial inventory discovery may take several minutes for large vCenters. A source that sits in Registering well beyond that points back to a Phase 1 prerequisite — most often NTP skew, a firewall gap, or insufficient role privileges.

Phase 3 Post-Registration Verification

Perform all of the following before declaring the registration successful.

UI checks
  • Sources page shows the new vCenter with status Online / Healthy.
  • Drill into the source and confirm the inventory tree populates (Datacenters → Clusters → Hosts → VMs / Datastores).
  • No red error banners or auth warnings on the source detail page.
  • Under source Settings, confirm the three options from Step 04 are toggled ON and no others.
vCenter side
  • In vCenter, confirm cohesitytestuser has an active session and was authorised against the expected role.
  • Check vCenter Events for any permission-denied entries against the Cohesity service account during the discovery window.
Connectivity smoke test (optional)
# From a Cohesity node (via support shell or iris_cli) — confirm reachability nc -vz testclust01.mgmt.testcloud.co.uk 443
Record keeping
  • Update the Backup Source register / CMDB with the new vCenter entry.
  • Log the change ticket reference against this runbook execution.
Rollback Rollback / De-registration
RB.1 De-register the vCenter source ROLLBACK Cohesity UI ▶

Back out the registration if the wrong vCenter was registered, the credential is incorrect, or the scope decision is reversed.

  1. Cohesity UI → Data Protection → Sources.
  2. Locate the vCenter, click the three-dot menu → Unregister.
  3. Confirm. Any Protection Groups referencing this source must be deleted or repointed first — Cohesity will block unregistration otherwise.
  4. Verify the source is no longer listed.
  5. In vCenter, optionally disable the cohesitytestuser session / remove the role binding if the credential was provisioned solely for this attempt.

Snapshots are not deleted. Unregistering a source does not delete the existing backup snapshots already on the cluster — those remain subject to their retention policy. If snapshots must also be purged, that is a separate, deliberate action with its own change approval.

⚠ Key Notes
  • NTP skew over 5 minutes between the cluster and vCenter causes SSO authentication to fail with a non-obvious error. Verify time sync before troubleshooting anything else.
  • Set the Step 04 toggles exactly: three ON, everything else OFF. Drift here changes backup-chain behaviour silently.
  • Use VM BIOS UUID to track VMs improves chain continuity but breaks on duplicate BIOS UUIDs — typically clones that did not regenerate their UUID. Resolve duplicates in vCenter before relying on it.
  • Auto Cancel Backups on low datastore space is a safety rail — always on. Set the threshold by percentage (~10% free), not GiB, so it auto-scales and handles multi-datastore disks.
  • Unregistration is blocked while any Protection Group still references the source — delete or repoint those Protection Groups first.
  • Unregistering a source does not delete existing snapshots; retention policy still applies. Purging snapshots is a separate, change-approved action.
  • Watch the service account for silent password rotation — keep the vault entry current, or Cohesity discovery will start failing without warning.
  • Creating Protection Groups, policies, and recovery jobs against this source is out of scope here — covered by a separate runbook.
