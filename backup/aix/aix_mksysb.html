AIX mksysb — Local, NIM, and Tape

Reference for capturing a bootable rootvg image (mksysb) on AIX as a local file, a NIM mksysb resource via nimsh, or a bootable tape — including pre-flight, verification, restore paths, and common failures.

REFERENCE root required
Scope
Bootable rootvg backup
Privilege
root
Output Targets
file · NIM · tape
Image Format
bosboot + backbyname
Includes
rootvg only
Restore Paths
bos_inst · alt_disk · restore -x

mksysb captures a bootable image of rootvg only — non-rootvg volume groups must be handled separately with savevg. The resulting image is bootable and can be consumed by NIM bos_inst, an alternate-disk install, or a tape boot from SMS. Three delivery methods are covered below: a local file under /backup, a remote push to a NIM master over nimsh, and a bootable tape on /dev/rmt1.

Reference Pre-flight checks
01 Validate rootvg, free space, OS level, and bosboot before snapshot REFERENCE ▶
# Confirm rootvg is the only VG being captured lsvg -l rootvg # Estimate image size — mksysb image is roughly used PPs * PP size lsvg rootvg | grep -E "USED PPs|PP SIZE" # Filesystem check — / and /usr should be clean and mounted df -g / /usr /var /tmp /home # OS level — record before snapshot oslevel -s instfix -i | grep ML lppchk -v # Check for prior errors that would invalidate the image errpt | head -20 # Confirm bosboot will succeed (mksysb runs bosboot internally) bosboot -a -d /dev/ipldevice

If lppchk -v returns anything, fix it before running mksysb. A snapshot of a broken filesets state will restore broken — the image is faithful to the running system, defects included.

Method 1 mksysb to local /backup file
02 1a · Prepare the /backup filesystem PREP ▶
# If /backup does not exist, create it in rootvg or a separate VG # Sized for at least 1.5x the rootvg used PPs to allow growth crfs -v jfs2 -g rootvg -m /backup -a size=20G -A yes -p rw mount /backup chmod 750 /backup # If /backup is already mounted, just verify free space df -g /backup

Plan for at least 1.5x the rootvg used-PP footprint. Image size is close to used capacity (not allocated), but headroom matters when -X auto-extension kicks in mid-run.

03 1b · Run mksysb to a timestamped image file CREATE ▶
# -i generate /image.data fresh (recommended — captures current LV layout) # -X auto-extend filesystem as needed during mksysb # -p do NOT pack files (faster, slight space cost) # -e exclude files listed in /etc/exclude.rootvg mksysb -i -X -p /backup/$(hostname)_$(date +%Y%m%d).mksysb

Typical runtime: 15–45 min depending on rootvg size and disk speed. The command runs bosboot internally, so the resulting file is bootable — the same image works for tape, NIM, and alt_disk_mksysb.

04 1c · Verify the image — TOC, volume info, spot-check REFERENCE ▶
# List image contents (table of contents) lsmksysb -lf /backup/$(hostname)_$(date +%Y%m%d).mksysb | head -30 # Verify volume group + image data lsmksysb -f /backup/$(hostname)_$(date +%Y%m%d).mksysb -B # Spot-check a known file (read-only, does not extract) restore -Tvqf /backup/$(hostname)_$(date +%Y%m%d).mksysb ./etc/passwd

A valid image will show image.data, bosinst.data, and a tar-format file tree. Empty / truncated TOCs almost always trace back to a filesystem unmount mid-run or an out-of-space condition.

Optional — exclude noisy filesystems via /etc/exclude.rootvg (one regex per line, anchored as ^./tmp/, ^./var/adm/ras/, ^./var/spool/lpd/) and add -e to the mksysb command. The exclude file is consulted only when -e is passed.

Method 2 mksysb to NIM master via nimsh
05 2a · Prerequisites — master, client, nimsh listener PREP ▶
# On the NIM master — confirm master is configured lsnim -l master # Confirm the client is defined as a NIM standalone object lsnim -l <client_name> # Filesystem with enough space for the image, e.g. /export/mksysb df -g /export/mksysb # On the client — nimsh must be enabled and running lssrc -s nimsh # If absent — convert client to nimsh communications and start it nimclient -C startsrc -s nimsh # Show nimsh listener (TCP 3901 / 3902) /usr/sbin/nimsh -l # Validate reachability from the master to the client over nimsh nim -o lslpp <client_name> | head

nimsh replaces the legacy rsh/rcmd mechanism — it operates over TCP 3901/3902 and does not require .rhosts. If the firewall between master and client blocks those ports, the nim -o define below will time out with 0042-001.

06 2b · Master-driven define (recommended) REMOTE EXEC ▶
# Run on the NIM master — the master invokes mksysb on the client over nimsh # and streams the image into the named resource path RES=mksysb_<client_name>_$(date +%Y%m%d) LOC=/export/mksysb/${RES} mkdir -p /export/mksysb nim -o define \ -t mksysb \ -a server=master \ -a location=${LOC} \ -a source=<client_name> \ -a mk_image=yes \ -a mksysb_flags="-i -X -p" \ ${RES}

This remotely runs mksysb on the client. Key attributes: source=<client_name> selects the client whose rootvg is captured; mk_image=yes tells NIM to actually run mksysb now (without it, define just registers an existing file as a resource); mksysb_flags is passed verbatim to the remote invocation. Confirm the client is quiesced before issuing — long-running databases or in-flight maintenance windows do not appreciate a 30-minute I/O burst.

07 2c · Alternate — client-initiated define REMOTE EXEC ▶
# Run on the client if you cannot get to the master directly # Same end result — the operation runs on the master via nimsh callback nimclient -o define \ -t mksysb \ -a server=master \ -a location=/export/mksysb/$(hostname)_$(date +%Y%m%d).mksysb \ -a source=$(hostname) \ -a mk_image=yes \ -a mksysb_flags="-i -X -p" \ mksysb_$(hostname)_$(date +%Y%m%d)

Useful when the master's CLI is gated behind a different bastion than the client. The resulting NIM resource is identical to the master-driven path — same name, same location, same attributes.

08 2d · Verify the resource on the NIM master REFERENCE ▶
# Confirm the NIM resource registered correctly lsnim -l ${RES} # File on disk should be non-zero and roughly match expected image size ls -l ${LOC} # Spot-check the image table of contents lsmksysb -lf ${LOC} | head

A healthy resource shows state = ready for use, a sensible alloc_count, and lsmksysb -lf returns a valid TOC. If state is anything else (e.g. not ready), the underlying file is absent or truncated — re-issue the define after fixing the cause.

09 2e · Use the resource for a future bos_inst REFERENCE ▶
# Network-install a target client from the mksysb resource nim -o bos_inst \ -a source=mksysb \ -a mksysb=${RES} \ -a spot=<spot_name> \ -a accept_licenses=yes \ -a boot_client=yes \ <target_client>

Cloning to a different machine is supported but requires post-install attention — verify hostname, IP, license acceptance, and any per-host application config that travels in rootvg.

Method 3 mksysb to tape
10 3a · Prerequisites and tape drive set to variable block size CAUTION ▶
# Drive present and configured? lsdev -Cc tape | grep rmt1 lscfg -vl rmt1 # Current block size — must be 0 (variable) for mksysb lsattr -El rmt1 -a block_size # If non-zero, set to variable chdev -l rmt1 -a block_size=0 # If the drive is in use: # chdev -l rmt1 -a block_size=0 -P # ODM only, takes effect at reboot

Variable block size (block_size=0) is mandatory. A fixed block size produces a tape that lsmksysb cannot read and SMS cannot boot — this is the single most common tape mksysb failure. Confirm with lsattr -El rmt1 -a block_size before every tape job; do not assume previous configuration persists across reboots if anyone has touched ODM.

11 3b · Rewind the tape with tctl PREP ▶
# Always rewind before writing — mksysb writes from BOT (beginning of tape). # A non-rewound tape will append at the wrong position or fail outright. tctl -f /dev/rmt1 rewind & # & returns the prompt while the rewind completes (LTO from end-of-tape: 30–90s) # Wait for the background job to finish before the next step wait

Other useful tctl operations on the same drive: status (drive busy/ready, block size, position), rewind (to BOT), offline (rewind and unload/eject), fsf N (forward-space N file marks), bsf N (back-space N file marks), weof N (write a file mark — rarely needed by hand).

12 3c · Run mksysb to /dev/rmt1 CREATE ▶
# -i generate fresh image.data # -X auto-extend any work filesystems # -p no pack # -e honour /etc/exclude.rootvg if present mksysb -i -X -p /dev/rmt1

mksysb writes four sections to tape, in order: (1) BOS boot image — kernel + minimal install kernel, (2) mkinsttape image — install scripts, (3) dummy TOC, (4) the actual rootvg backup in backbyname format. This four-section layout is what makes the tape bootable from SMS, and is also why a single-file extract requires fsf 3 (skip three file marks to reach section 4).

Typical runtime: tape-speed limited. LTO-7/8 native ~300 MB/s, so a 60 GB rootvg lands around 4–8 minutes plus tape mechanics (load, BOT seek, write, file marks).

13 3d · Verify the tape — TOC and spot-check REFERENCE ▶
# Rewind first — verification reads from BOT tctl -f /dev/rmt1 rewind # Show table of contents from tape lsmksysb -lf /dev/rmt1 | head -40 # Spot-check a known file (read-only) restore -Tvqf /dev/rmt1 ./etc/passwd

A valid tape returns the same image.data / bosinst.data / file tree as a file-based image. If lsmksysb errors with 0511-… block-size or read errors, the drive was not in variable block mode at write time — re-check lsattr -El rmt1 -a block_size and recreate the tape from scratch.

14 3e · Eject and label REFERENCE ▶
# Rewind and unload tctl -f /dev/rmt1 offline

Label with: hostname, OS level (oslevel -s), date, and rootvg used size. Store with the corresponding bosinst record / customer ticket — without this metadata, restore-side decisions (which tape, which level, what's missing) become guesswork.

15 3f · Restore from tape — full rootvg or single file REFERENCE ▶

Boot from tape (full rootvg restore): insert tape, power-cycle into SMS (key 1 / 5 depending on platform), then Select Boot Options → Install/Boot Device → Tape → select the rmt drive. Tape boots into the BOS install menus — pick Install with Settings and follow the migration / overwrite path.

# Restore a single file from tape tctl -f /dev/rmt1 rewind # mksysb image is the 4th file on tape — skip the first 3 file marks tctl -f /dev/rmt1 fsf 3 restore -xvqf /dev/rmt1 ./path/to/file

fsf 3 is non-negotiable for tape extracts — sections 1–3 (boot image, mkinsttape, dummy TOC) must be skipped to land on the backbyname data. Forgetting this gives a "garbage" read because restore tries to interpret the boot image as a backup archive.

⚠ Key Notes
  • mksysb captures rootvg only. Non-rootvg VGs (application data, database volumes, user home filesystems on separate VGs) need savevg — a "successful" mksysb that misses the apps is the most-common post-restore complaint.
  • /etc/exclude.rootvg is honoured only when -e is passed. The file uses regex against the full path with paths anchored as ^./tmp/; a missing leading ^./ usually means the rule does nothing.
  • NIM define needs mk_image=yes to actually capture a fresh image. Without it, nim -o define -t mksysb only registers an existing file as a resource — easy to misdiagnose because the command returns success either way.
  • Tape mksysb requires block_size=0 (variable) on rmt1. A fixed block size produces a tape that lsmksysb cannot read and SMS cannot boot — the single most common tape gotcha.
  • A tape mksysb writes four sections in order: BOS boot, mkinsttape, dummy TOC, and the actual backbyname archive. Single-file extracts require tctl ... fsf 3 to skip the first three file marks before restore -x.
  • Always tctl ... rewind before writing or verifying a tape. mksysb writes from BOT — a non-rewound tape appends at the wrong position or errors out.
  • Run lppchk -v before every snapshot. A mksysb of a broken-fileset state restores broken; the image is faithful to the running system, defects included.
