01 The generator
01 build_index.py walks the repo and regenerates two files GENERATOR ▶

The script globs *.html, reads <title>, <meta name="description">, keywords, and reviewed from each, groups by top-level folder (the category) with the second-level folder as the sub-tag, and writes both the landing page and the sidebar's nav tree.

python3 scripts/build_index.py # Wrote index.html, N runbooks across M categories # Wrote assets/js/nav-data.js, sidebar rail nav tree

It skips .git/, .github/, node_modules/, scripts/, docs/, assets/, _sandbox/, and index.html itself. Categories, their display name, accent, and order live in the CATEGORIES dict near the top; the page HTML lives in the STYLES, SCRIPT, and TEMPLATE string constants lower down.

Also a linter: the build prints a warning for any runbook unreviewed for 12+ months or missing a reviewed tag. It warns rather than fails; a stale runbook is a documentation problem, and failing the build would just stop the wiki updating.

02 CI runs it on push CI ▶

.github/workflows/build-index.yml runs the script on every push that touches an HTML file or the build script, then commits the regenerated index.html and nav-data.js back with [skip ci] in the message, so there is no loop. A new runbook becomes a card and a rail entry within about 30 seconds of the push.

Never hand-edit index.html or nav-data.js. Both are regenerated on every push and manual edits are overwritten. Change the landing page through the template constants in build_index.py.

02 The asset layer

Every page loads the same five shared files. This is where the design language and behaviour live, so a runbook is almost entirely content.

FileResponsibility
css/tokens.cssPalette, four-font stack, reset, base body, keyframes, and the dark theme (single token swap)
css/runbook.cssShared components: crumb, title, badges, identity strip, env, phase label, steps, code panel, notes, warn-box, tables, footer
css/sidebar.cssThe category rail, its filter, and the top-left control pill; collapse on desktop, drawer below 1080px

js/nav-data.jsGenerated nav tree; chrome.js reads window.WIKI_NAV to build the rail
js/boot.js + css/boot.cssLanding page only: the IPL boot sequence and hidden shell
03 What chrome.js does at load
03 One script wires every shared behaviour RUNTIME ▶

Because chrome.js does all of this, runbooks carry no panel markup, no copy script, and no nav markup of their own.

On loadEffect
Code panelsWraps every .cmd in a header bar (mono language label left, copy-icon button right). data-lang overrides the label, default shell.
StepsBinds the accordion toggle on each .step-header and adds a hover-revealed # deep-link anchor per step; a linked step opens and flashes on arrival.
Reviewed cellAppends the third identity-strip cell from <meta name="reviewed">, colour-coded by age.
Sidebar railBuilds the rail from window.WIKI_NAV, with its own filter. Works out the site root from its own src, so links resolve from any folder depth.
Control pillHome, sidebar toggle, and theme toggle. Persists theme (wiki-theme) and rail state (wiki-rail) in localStorage, shared across every page.
04 The identity-strip model
04 Fixed identity strip, variable environment block LAYOUT ▶

Every runbook opens with a three-cell .meta.meta-identity strip (Runbook ID · Operation · Last reviewed) that is identical in shape everywhere, followed by an .env block whose cells vary per runbook. The strip is the constant a reader learns to scan; the environment block is the per-procedure detail. The reviewed cell is authored nowhere, chrome.js appends it, so the shape can't drift.

The full history of moving from bespoke per-runbook meta grids to this fixed strip is in docs/META_GRID_MIGRATION.md. Both layouts render correctly, so the migration could stop half-way without breaking anything.

05 The IPL boot landing
05 The landing page boots like a pSeries box, once EASTER EGG ▶

boot.js (landing page only) plays an AIX / pSeries IPL sequence, operator-panel LED checkpoints, the boot banner, rc.boot device configuration, then reveals the index. It runs once ever per browser, tracked in localStorage, and prefers-reduced-motion skips it entirely.

TriggerResult
#ipl in the URLReplay the full IPL on demand
"#_" in the headerDrops into an interactive maintenance shell
help, then cat DEVNOTESThe payoff, developer notes and TODOs
reboot (in the shell)Plays the IPL again

The shell's "filesystem" and the DEVNOTES text live at the top of boot.js, flagged EDIT ME, swap in whatever you want people to find. Keep it ASCII so it stays on-brand.

06 Working on it locally
06 Rebuild, serve over http, check in a private window PREVIEW ▶
python3 scripts/build_index.py # regenerate index + nav-data ./serve.sh # python3 -m http.server on :8765 open http://localhost:8765/

Never open a runbook from Finder. file:// blocks the shared stylesheets in assets/css/ from loading (they live in a parent directory) and the page renders unstyled. Serve over http. And check in a private window, chrome.js is a stable URL that keeps gaining behaviour, so a cached copy will make a working change look broken.

⚠ Key Notes
  • Hand-built static HTML, no framework, no runtime build step, just build_index.py, the shared assets, and a browser.
  • index.html and nav-data.js are generated. Edit the template constants in build_index.py, never the output.
  • The palette is fixed at five families, stone paper, ochre, rust, sage, code panel. Adding a sixth colour always looks wrong.
  • Reviewed dates come from <meta name="reviewed">, not git. A cosmetic commit must never make a stale runbook look fresh.
  • Serve over http and verify in a private window before pushing.
