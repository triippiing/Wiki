TL;DR: Save the file under <category>/<subtag>/your-runbook.html, give it a <title>, a <meta name="description">, and a <meta name="reviewed">, link the shared assets, open with the identity strip, then commit and push to main. The card and the sidebar entry appear within about 30 seconds.

Phase 1 Where the file goes

Place the runbook in one of the category folders. The top-level folder is the category; the second-level folder becomes the card's sub-tag, shown as aix · lvm.

CategoryPathIndex accent
AIXaix/ochre

Backupbackup/rust
Cohesitycohesity/faint

VTLvtl/ink

Metameta/ink-2

File naming: URLs work either way, but hyphens beat spaces for sharing. aix-rootvg-disk-replacement.html is nicer than AIX rootvg disk replacement.html, which resolves as ...AIX%20rootvg%20disk%20replacement.html.

Phase 2 Required head metadata
01 The three tags the build reads REQUIRED ▶
<title>Your Runbook Title</title> <meta name="description" content="One-sentence summary that becomes the card body."> <meta name="reviewed" content="2026-07-19">

build_index.py reads all three. Without <title> it falls back to the filename. Without a description the card has no body paragraph. Without a review date the card reads UNREVIEWED and the build prints a warning.

02 Optional: searchable keywords OPTIONAL ▶
<meta name="keywords" content="vg lv jfs2 mklv crfs extendvg">

The landing page's live search checks title, category tag, filename, and these keywords. Add terms a reader might search for that aren't already in the title.

03 Minimum boilerplate TEMPLATE ▶
<!DOCTYPE html> <html lang="en"> <head> <meta charset="UTF-8"> <meta name="viewport" content="width=device-width, initial-scale=1.0"> <title>Your Runbook Title</title> <meta name="description" content="One-sentence summary."> <meta name="reviewed" content="2026-07-19"> <meta name="keywords" content="optional space-separated search terms"> <link rel="preconnect" href="https://fonts.googleapis.com"> <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin> <link href="https://fonts.googleapis.com/css2?family=IBM+Plex+Mono:wght@400;600&family=Public+Sans:wght@400;500;600&family=Source+Serif+4:opsz,wght@8..60,600&family=Spectral:wght@600&display=swap" rel="stylesheet"> <link rel="stylesheet" href="../../assets/css/tokens.css"> <link rel="stylesheet" href="../../assets/css/runbook.css"> <link rel="stylesheet" href="../../assets/css/sidebar.css"> <script src="../../assets/js/nav-data.js" defer></script> <script src="../../assets/js/chrome.js" defer></script> <style> /* bespoke classes only */ </style> </head> <body> <!-- runbook content --> </body> </html>

Asset depth: runbooks two folders deep (aix/lvm/foo.html) link with ../../. A page one folder deep (vtl/foo.html, meta/foo.html) uses ../. Load tokens.css first, runbook.css and sidebar.css depend on its variables. Load nav-data.js before chrome.js; both are defer, which preserves order.

Phase 3 The identity strip and environment block

Every runbook opens with the same two blocks, in this order. This shape is live on every runbook and new ones should match it.

04 Identity strip, three cells, always the same shape STRUCTURE ▶

You author two cells. The third, Last reviewed, is appended at load by chrome.js from <meta name="reviewed">, so the markup never writes it.

<div class="meta meta-identity"> <div class="meta-cell"><div class="meta-label">Runbook ID</div><div class="meta-val mono">RB-PERF-AIX-001</div></div> <div class="meta-cell"><div class="meta-label">Operation</div><div class="meta-val info">Performance triage · CPU / memory / I/O</div></div> <!-- Last reviewed cell appended at load by chrome.js --> </div>
05 Environment block, everything else, per runbook STRUCTURE ▶

The .env block holds hosts, OS levels, NIM masters, windows, rollback method, spare-disk requirements, whatever this procedure needs confirmed before you start. Its contents vary per runbook, and that is correct. It sits on a quieter surface so it reads as secondary to the identity strip.

<div class="env"> <div class="env-head">Environment</div> <div class="env-grid"> <div class="env-cell"><div class="env-label">Scope</div><div class="env-val info">Bottleneck triage</div></div> <div class="env-cell"><div class="env-label">Privilege</div><div class="env-val warn">root for kernel tools</div></div> <div class="env-cell"><div class="env-label">Primary Tools</div><div class="env-val">topas · vmstat · iostat</div></div> </div> </div>

Value cells take the same semantic variants in both blocks: info (ochre), ok (sage), warn (rust), mono.

06 Runbook IDs CONVENTION ▶

The scheme is RB-<DOMAIN>-<PLATFORM>-<NNN>. Existing allocations live in scripts/runbook_meta.py, read it to pick the next free number in your domain, then write the ID into the identity strip yourself. That file is a record of what has been allocated, not a live source; editing a string there does not propagate into any page.

Keep the RB-ID out of the <title> and the <h1>. Both surface in browsers, search, and the index card, and the ID is noise in all three. It belongs in the identity strip and the footer. Keep the Operation line short, roughly 40 characters, the fuller description already appears as the page subtitle.

Phase 4 Last reviewed
07 The one date a reader can trust IMPORTANT ▶

Bump this only when you have actually checked the procedure is still correct. Not for a typo, and not when a sweeping change (a restyle, a new shared component) touches the file. It only means anything if it means "a human confirmed this still works." It is deliberately not derived from git: a cosmetic commit across every runbook would reset every date at once and make stale runbooks look freshly checked.

Don't hand-edit the tag. Use the helper, it stamps the date and regenerates the index in one go:

python3 scripts/mark_reviewed.py aix/lvm/aix_lvm_basics.html # stamp today python3 scripts/mark_reviewed.py aix/administration/*.html # several at once python3 scripts/mark_reviewed.py --date 2026-06-01 vtl/*.html # a specific day python3 scripts/mark_reviewed.py --stale # what needs a look

chrome.js renders it as the third identity-strip cell, colour-coded by age: sage under 6 months, ochre 6 to 12, rust beyond 12 or if absent. Commit the runbook, index.html, and nav-data.js together.

Phase 5 Styling, shared vs bespoke
08 The asset layer does the heavy lifting ASSETS ▶
FileWhat it carries
css/tokens.cssThe stone / ochre palette, the four-font stack, reset, base body styles, keyframes, and the dark theme. Edit to shift the design across every runbook at once.
css/runbook.cssShared components: crumb, page title, badges, wrap, identity strip, env block, phase label, step accordion, code panel, note variants, warn-box, tables, footer.
css/sidebar.cssThe 212px category rail and its filter, plus the top-left control pill. Collapses on desktop, becomes a drawer below 1080px.
js/chrome.jsBuilds code-panel headers and copy buttons, binds step accordions and deep links, appends the reviewed cell, injects the rail, and handles theme + rail state.
js/nav-data.jsGenerated. The nav tree the rail is built from. Never edit by hand.

Runbooks do not hand-write code-panel markup, a copy script, or any nav markup, chrome.js builds all of it around each .cmd at load. Set data-lang on a .cmd to override the language label, which defaults to shell.

09 Keep bespoke CSS to content-shaped classes only DISCIPLINE ▶

An inline <style> block is only for classes specific to your runbook, usually content-shaped tables or step-accent overrides:

.step.ph-cpu { border-left-color: var(--rust); } .step.ph-disk { border-left-color: var(--ochre); }

Always reference token variables (var(--ochre), var(--border)). Never hard-code colours. The palette is deliberately restrained: stone paper, one ochre accent, rust for caution, sage for verify, mono for anything machine-readable. If you catch yourself copy-pasting a bespoke component into a second runbook, promote it into runbook.css instead.

Reference cheatsheets are an intentional exception. Pages under reference/ use a different, denser lookup-card design on purpose. Model a new cheatsheet on the existing two, not on the runbook library. A procedural runbook placed in reference/ still uses the standard layout.

Phase 6 Adding a new category
10 Give a new folder a name, order, and accent CONFIG ▶

Drop a runbook into a new folder and it appears under a section with a neutral accent. To name and colour it, add an entry to the CATEGORIES dict near the top of scripts/build_index.py:

CATEGORIES = { # ...existing categories... "database": {"name": "Database", "accent": "ref", "order": 9}, }

order is the position on the landing page. To add a fresh accent, define an .accent-<name> rule in the STYLES constant lower in the same file, pointing at a token colour, exactly how the Meta section you're reading was added (.accent-meta { background: var(--ink-2); }).

Phase 7 Build and test locally
11 Regenerate, then serve over http PREVIEW ▶
# regenerate index.html + assets/js/nav-data.js python3 scripts/build_index.py # serve over http, do NOT open from Finder (file:// blocks ../assets) ./serve.sh open http://localhost:8765/

Check pages in a private window. chrome.js is a stable URL that keeps gaining behaviour, so a browser will happily serve a cached copy and you'll conclude a change is broken when it isn't. Cmd+Option+R ("Reload From Origin") also works.

⚠ Key Notes
  • Never edit index.html or assets/js/nav-data.js by hand, both are regenerated on every push and manual edits are overwritten. Change the landing page via the STYLES / SCRIPT / TEMPLATE constants in build_index.py.
  • Bump reviewed only when a human has confirmed the procedure still works. It is the one field a reader is meant to trust.
  • Keep the RB-ID in the identity strip and footer, never in the <title> or <h1>.
  • Load tokens.css first; load nav-data.js before chrome.js.
  • Reference token variables in bespoke CSS. Never hard-code a colour, and don't add a sixth accent to the palette.
  • Serve over http and check in a private window, never open a runbook from file://.
